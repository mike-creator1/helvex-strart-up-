Tutorial format

Every tutorial follows the same shape.

Four sections, in this order, every time. The structure is on purpose — by the time you have read three tutorials, the next one is faster, and you know where to skim if you only need one part.

  1. Section 01

    Problem

    The work the tutorial is for. A specific job — not a category. "Reply to inbound support tickets with the right runbook" is a problem. "Customer service" is not. The constraints, the data, the success bar are all on the page.

  2. Section 02

    Model choice

    Which of the three we picked — Ether, Nexus, or Prometheus — and why. The reasoning is in the open: latency budget, depth required, tools needed, cost ceiling. If we changed our mind during the work, that change is in the write-up too.

  3. Section 03

    Workflow

    The system prompt, the tools, the gates, the eval set. Copy-and-pasteable. The point is that you can have the same result in your own workspace by following along — not that you watch a demo and trust it.

  4. Section 04

    Debug

    What went wrong on the way. Hallucinations, refusal mis-calibration, tools that timed out, prompts that drifted. The fixes that worked, the fixes that didn't, and the places we left rough edges for the next person to improve.
